Claude Code 调优方案

从全局规则、模块化配置、记忆系统到自动化脚本，系统性提升 Claude Code 的工程效率与输出质量。新环境 30 分钟内完成全套部署。

核心问题 四层架构 快速上手 核心能力 效果数据 实施路径 CC-Switch

核心问题

Claude Code 的效率瓶颈不在模型能力，而在上下文管理：

问题表现根因
Token 浪费一次会话消耗 80K-200K Token，大部分是冗余输出CLI 命令输出未过滤、响应风格冗长
上下文腐烂长会话后 Claude 开始"犯糊涂"、矛盾、遗忘超过 120K Token 后检索准确率显著下降
需求偏差Claude 做的不是你想要的缺少实现前的需求对齐机制
无法复用每次新会话从零开始缺少跨会话记忆和项目知识沉淀

四层优化架构

架构逻辑：能用 → 做对 → 高效 → 工程化，逐层递进，每层依赖下层。

L4

工程化层 — "规模化协作" L4

自定义 Commands、角色型智能体、Skills 技能、MCP 服务器 — 让 Claude Code 成为团队的工程化平台。

  • 自定义 Commands：/review / /fix-issue / /deploy / /test-plan，复用高频操作
  • 7 个角色型智能体：UI 设计师 / 前端架构师 / 代码审查员 / 安全扫描器 / 性能分析器 / API 测试 / 文档生成，按需调用，上下文隔离
  • 13 个 Skills：mattpocock/skills + addyosmani/agent-skills，覆盖全生命周期 Define → Plan → Build → Verify → Review → Ship
  • MCP 服务器：Filesystem / GitHub，扩展外部能力
L3

效率层 — "用更少 Token 做更多事" L3

Token 是 Claude Code 最核心的成本和性能瓶颈，效率层聚焦 Token 优化。

  • rtk：CLI 输出过滤，减少 50-90% Token
  • caveman：响应风格压缩（Lite/Full/Ultra 三档），减少 65-87% Token
  • ccusage：用量基线分析（daily/monthly/breakdown）
  • Claude Monitor：实时限额预警，避免意外超额
L2

行为约束层 — "让 Claude 做对事" L2

通过结构化的规则和守卫机制，确保 Claude 在每个项目中都能正确理解并遵守规范。

  • CLAUDE.md：全局 + 项目分层配置，定义行为准则和项目上下文
  • Rules 模块化：.claude/rules/，按 glob 匹配自动生效（通用 3 个 + 技术栈 6 个）
  • Hooks 守卫：PreToolUse / PostToolUse / Stop，确定性守卫机制
  • 权限白名单：/permissions，减少重复确认，提升交互效率
L1

基础层 — "能跑起来" L1

环境就绪是优化的前提，基础层确保 Claude Code 稳定运行。

  • Claude Code CLI + CC-Switch（多账号模型切换）
  • Node.js + Git + PowerShell 7
  • 网络 / 代理 / 密钥配置

核心能力

1. CLAUDE.md 分层配置

Claude Code 支持多级 CLAUDE.md，按作用域自动叠加加载。

第一层：全局行为准则（用户级）

放置在 ~/.claude/CLAUDE.md，只放通用行为约束，不超过 50 行。融合 Karpathy 四原则：

#准则说明
1思考先行不确定时主动提问，不假设。存在多种实现方式时列出权衡
2简洁优先只做被要求的功能，不添加"灵活性"和"可扩展性"。200 行能缩到 50 行就重写
3精准变更只改必须改的，不顺手"改进"旁边的代码。你的变更产生的孤立代码必须清理
4目标驱动将任务转化为可验证的目标，用测试循环验证直到通过
5输出与边界响应精简，不添加未被要求的注释，不引入未使用过的第三方库
6语言偏好对话/注释/输出使用简体中文，commit message 保持英文

第二层：项目专属配置（项目级）

在项目根目录创建 CLAUDE.md，只写项目专属内容，不重复全局已有的行为准则：

  • 项目信息：目标、技术栈、成功指标
  • 目录结构：关键目录说明
  • 常用命令：安装/启动/测试/构建/Lint
  • 编码规范：语言、测试、提交格式
  • Token 管理建议

2. Rules 模块化

按职能拆分规则文件，通过 glob 匹配自动生效，支持通用基线 + 技术栈专用叠加。

通用基线规则（3 个）

文件glob 匹配核心规则
frontend.mdfrontend/**, src/components/**组件单一职责 ≤200 行 · TypeScript strict · CSS Modules/Tailwind · 优先复用公共组件
backend.mdbackend/**, src/api/**, src/services/**API 输入验证 · AppError 统一错误 · 事务保护 · 外部调用超时重试
testing.md**/*.test.*, **/*.spec.*describe > it > expect · 测试独立 · Mock 外部依赖 · 覆盖正常+边界+错误

3. Hooks 守卫

Hooks 是"保证型机制"——提示只是建议，Hook 是保证。

PreToolUse — 拦截危险命令

在 Bash 命令执行前拦截，匹配到以下模式则阻断：

  • rm -rf / — 递归删除根目录
  • git push --force — 强制推送
  • DROP TABLE — 删除数据库表
  • format [a-z]: — 格式化磁盘

PostToolUse — 文件修改提醒

Write / Edit 工具执行后，提醒运行 lint/test 确认变更质量。

Stop — 验收检查

Claude 停止响应前，提醒输出应包含：变更摘要、验证结果、风险点。

4. 自定义命令

自定义命令是高频操作的快捷入口，文件名即命令名。~/.claude/commands/ 全局生效，.claude/commands/ 仅项目生效。

命令级别用途可用工具
/review全局审查当前分支代码变更，按 Critical/High/Medium/Low 分类Read, Bash(git:*)
/fix-issue全局分析并修复 GitHub Issue，自动关联 commitBash(gh:*), Read, Write
/deploy项目部署前检查清单（测试/lint/环境变量/迁移/回滚）Bash(npm:*), Bash(git:*)
/test-plan项目根据需求生成结构化测试计划 + 测试文件骨架Read, Glob, Grep, Write

上下文窗口规则

理解上下文窗口对输出质量的影响，是 Token 管理的基础。

最佳性能
< 120K Token
质量下降
120K - 500K
不推荐
500K+（成本翻倍）

关键阈值

  • 120K 以下：模型保持最佳性能
  • 120K - 500K：质量开始下降，建议主动 /compact
  • 500K+：输出质量可能不如 200K 的会话，但成本翻倍

建议：每 15-20 条消息检查一次 /context，在 60% 阈值时主动使用 /compact。

会话管理原则

核心原则：一个会话只做一件事，话题切换时开新会话。

推荐会话拆分

会话 1: 需求分析 + PRD（~30K Token）→ /compact → 继续
会话 2: 架构设计（新会话，/clear）
会话 3: 前端实现（新会话，/clear）
会话 4: 后端实现（新会话，/clear）
会话 5: 联调测试（新会话，/clear）

每个 Provider 内以下字段保持同厂商一致：

  • ANTHROPIC_MODEL
  • ANTHROPIC_SMALL_FAST_MODEL
  • ANTHROPIC_DEFAULT_HAIKU_MODEL / SONNET_MODEL / OPUS_MODEL
  • CLAUDE_CODE_SUBAGENT_MODEL

切换后固定执行

# 1. 新开终端
# 2. 验证状态
claude  # 进入后输入 /status

常见错误

  • 当前 Provider 是 A，但模型填了 B 厂商模型名
  • 只改了 ANTHROPIC_MODEL，未同步默认模型字段
  • 切换后未重开会话，仍使用旧配置缓存
