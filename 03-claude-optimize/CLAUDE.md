## 行为准则

### 1. 思考先行（Think Before Coding）

**不假设。不隐藏困惑。暴露权衡。**

- 不确定时主动提问，不假设——声明你的假设
- 存在多种实现方式时，列出各方案的权衡，不偷偷选一种
- 有更简洁的方案就主动提出，必要时反驳不合理需求
- 遇到困惑立即停下，明确说出哪里不清楚

### 2. 简洁优先（Simplicity First）

**最小代码解决问题。不投机。**

- 只做被要求的功能，不添加"灵活性"和"可扩展性"
- 一次性代码不做抽象
- 不为不可能的场景写错误处理
- 如果 200 行能缩到 50 行，重写它
- 测试标准：高级工程师会觉得过度设计吗？如果是，简化

### 3. 精准变更（Surgical Changes）

**只改必须改的。只清理自己制造的垃圾。**

- 不顺手"改进"旁边的代码/注释/格式
- 不重构没坏的东西，匹配现有风格（即使你会写得不同）
- 你的变更产生的孤立代码（无用 import/变量/函数）必须清理
- 预存在的死代码只报告，不删除（除非被要求）
- 测试标准：每一行变更都能追溯到用户的请求

### 4. 目标驱动（Goal-Driven Execution）

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标：
- "加验证" → "为非法输入写测试，然后让测试通过"
- "修 bug" → "写复现测试，然后让测试通过"
- "重构 X" → "确保重构前后测试都通过"

多步任务先列计划：
```
1. [步骤] → 验证: [检查点]
2. [步骤] → 验证: [检查点]
3. [步骤] → 验证: [检查点]
```

强目标让 AI 独立循环，弱目标（"让它能跑"）则需要反复确认。

### 5. 输出与边界

- 响应精简，避免重复叙述
- 代码变更后给出：变更摘要、验证结果、风险点
- 不添加未被要求的注释
- 不修改未被明确要求的文件
- 不引入未在项目中使用过的第三方库
- 不在代码中硬编码密钥或敏感信息

### 6. 语言偏好

- 所有对话、解释、提示信息使用简体中文
- 代码注释使用简体中文（除非项目已有英文注释规范）
- 命令行输出说明使用简体中文
- 技术术语可在首次出现时附英文原文，后续统一用中文
- commit message 可保持英文（遵循 Conventional Commits 规范）

## 项目上下文

### 项目信息
- 项目目标：Claude Code 可复用调优方案——覆盖全局配置、项目模板、部署脚本、技能/智能体/规则体系
- 技术栈：Markdown 文档 + PowerShell 部署脚本 + JSON 配置
- 成功指标：新环境 30 分钟内完成全套配置部署，配置文件 100% 可拷贝复用

### 目录结构
- `.claude_global/`：全局配置模板（部署到 `~/.claude/`）
- `templates/`：项目级模板（复制到新项目根目录）
- `.claude/`：当前项目的工作配置（与 templates/ 同步）
- `scripts/`：部署脚本（setup-claude.ps1、reinstall-claude-to-cc-switch.ps1）

### 常用命令
- 部署全部：`.\scripts\setup-claude.ps1 -Phase all -ProjectDir "D:\target"`
- 部署阶段一：`.\scripts\setup-claude.ps1 -Phase 1`
- 部署阶段二：`.\scripts\setup-claude.ps1 -Phase 2 -ProjectDir "D:\target"`

### 同步规则
- `.claude/` 是工作目录，`templates/` 是模板目录，两者必须保持同步
- 修改 .claude/ 中的文件后，需同步到 templates/ 对应位置
- 修改 templates/ 中的文件后，需同步到 .claude/ 对应位置
- `.claude_global/` 独立维护，仅包含全局级配置

### 编码规范
- 文档语言：简体中文
- 配置文件：JSON（settings.json、.mcp.json）
- 命令/规则/智能体：Markdown（含 YAML frontmatter）
- 脚本：PowerShell 7（pwsh）

### Token 管理
- 响应尽量精简
- 话题切换时建议 /clear 开启新会话
