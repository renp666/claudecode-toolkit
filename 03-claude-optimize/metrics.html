上下文窗口规则

理解上下文窗口对输出质量的影响，是 Token 管理的基础。

最佳性能
< 120K Token
质量下降
120K - 500K
不推荐
500K+（成本翻倍）

关键阈值

  • 120K 以下：模型保持最佳性能
  • 120K - 500K：质量开始下降，建议主动 /compact
  • 500K+：输出质量可能不如 200K 的会话，但成本翻倍

建议：每 15-20 条消息检查一次 /context，在 60% 阈值时主动使用 /compact。

原生命令速查

命令用途何时使用
/context查看上下文占用每 15-20 条消息检查一次
/rewind回退到之前的对话点发现方向错误时
/compact手动压缩上下文在 60% 阈值时主动使用
/clear清空会话重新开始切换不相关话题时
/cost查看当前消耗关键节点检查
/model切换模型简单任务降级到 Sonnet
/resume恢复之前的会话需要继续之前的工作时

会话管理原则

核心原则：一个会话只做一件事，话题切换时开新会话。

推荐会话拆分

会话 1: 需求分析 + PRD（~30K Token）→ /compact → 继续
会话 2: 架构设计（新会话，/clear）
会话 3: 前端实现（新会话，/clear）
会话 4: 后端实现（新会话，/clear）
会话 5: 联调测试（新会话，/clear）

工具效果对比

rtk — CLI 输出过滤

仅对 Bash 工具调用生效，Claude 内置工具（Read/Grep/Glob）不经过 rtk。

  • 过滤 git status / git log / npm install 等命令的冗余输出
  • 减少 50-90% 的 Token 消耗
  • 安装：rtk init -g

caveman — 响应压缩

三档压缩强度，安全场景自动降级：

  • Lite：保留语法，减少冗余修饰
  • Full：省略冠词，压缩句子结构
  • Ultra：电报式，仅保留核心信息

安全边界：安全审计、客户文档、法律合规场景不压缩。代码块、技术术语、错误信息始终保持原文。

ccusage — 用量监控

没有基线数据，所有优化都是盲人摸象。

npx ccusage@latest daily              # 每日用量
npx ccusage@latest monthly --breakdown # 月度成本分布

Claude Monitor — 实时预警

实时监控 Token 用量，接近限额时预警。

pip install claude-monitor
claude-monitor --plan max5    # Max5 计划
claude-monitor --plan max20   # Max20 计划
