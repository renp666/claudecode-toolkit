## 行为准则

### 1. 思考先行（Think Before Coding）

**不假设。不隐藏困惑。暴露权衡。**

- 不确定时主动提问，不假设——声明你的假设
- 存在多种实现方式时，列出各方案的权衡，不偷偷选一种
- 有更简洁的方案就主动提出，必要时反驳不合理需求
- 涉及国产大模型 API 配置时，优先使用官方文档作为权威依据

### 2. 简洁优先（Simplicity First）

**最小代码解决问题。不投机。**

- 只做被要求的功能，不添加"灵活性"和"可扩展性"
- 配置模板（config.toml）保持最简，不预设未来可能用不到的选项
- 文档说明直击痛点，不堆砌无关背景

### 3. 精准变更（Surgical Changes）

**只改必须改的。只清理自己制造的垃圾。**

- 新增国产模型接入说明时，保持与其他模型的格式一致
- 变更后产生的孤立引用/死链接必须同步修复
- 预存在的过时信息只标注，不删除（除非被要求）

### 4. 目标驱动（Goal-Driven Execution）

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标：
- "接入 DeepSeek" → "在 Codex++ 管理工具中完成供应商配置，能用 Codex 正常对话"
- "更新命令清单" → "新的常用命令已加入速查表，格式与现有条目一致"
- "修复配置问题" → "确认报错消失，对应 FAQ 条目已更新"

### 5. 输出与边界

- 响应精简，避免重复叙述
- 所有配置模板必须经过可用性验证（至少在一种环境测试过）
- 不引入未经验证的第三方工具/插件，除非明确标注"社区测试"
- 不在配置中硬编码 API Key 或敏感信息，所有 Key 用占位符表示
- 信息溯源：所有外部数据标注来源和可信度等级

### 6. 语言偏好

- 所有对话、解释、提示信息使用简体中文
- 代码注释使用简体中文
- 技术术语首次出现时附英文原文，后续统一用中文
- commit message 可保持英文（遵循 Conventional Commits 规范）

## 项目上下文

### 项目信息
- 项目目标：Codex++ 桌面版 + 国产大模型完整落地指南——覆盖安装、配置、使用、生态全流程
- 技术栈：Markdown 文档
- 成功指标：
  - 新手按照文档 15 分钟内完成首个国产模型接入
  - 故障排查清单覆盖 80% 的常见问题

### 目录结构
- `CLAUDE.md`：项目行为准则（本文档）
- `Codex++桌面版-国产大模型落地指南.md`：主文档
- `常用命令.md`：Codex++ 常用操作速查表
- `分析-可行性评估.md`：国产模型方案评估分析
- `01templates-模板文件目录/`：项目模板文件

### 方案核心三要素

| 要素 | 说明 | 交付物 |
|------|------|--------|
| **适配方案** | Codex++ 桌面增强器（主方案） | 主文档第 1 章 |
| **模型配置** | DeepSeek / Qwen / GLM / Kimi 接入 | 主文档第 2 章 |
| **生态资源** | 代理工具 / 插件 / Skills 汇总 | 主文档第 5 章 |

### 编码规范
- 文档语言：简体中文
- 模板：遵循各模型官方文档的最新格式

### 维护原则
- 版本号同步：每次重大更新在文档头标注版本日期
- 实测优先：新增模型接入说明前至少在一台 Windows 环境下验证
- 弃用标注：发现过时信息时，打上 ⚠️ 标记并标注替代方案

---

**验证标准：** 如果文档中的配置步骤无法按图索骥完成操作，或关键链接失效——说明需要更新。
